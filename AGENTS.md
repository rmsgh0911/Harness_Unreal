# AGENTS.md

이 저장소는 Unreal Engine 프로젝트다.

## 작업 전 읽기

1. 프로젝트 루트의 `README.md` 또는 현재 작업 상태 문서가 있으면 먼저 확인한다.
2. `Harness/README.md`, `Harness/state.md`, `Harness/next.md`가 있으면 확인한다.
3. 오늘 날짜의 `Harness/cycles/YYYY-MM-DD.md`가 있으면 확인한다.
4. 멀티 에이전트 리뷰나 전환이 필요하면 `Harness/config/agents.json`, `Harness/config/cycle_policy.json`을 확인한다.
5. 현재 요청에 필요한 `Source/`, `Config/`, `Plugins/`, `Content/` 파일만 추가로 확인한다.

## 기본 목표

사용자가 기능을 지시하면 에이전트는 가능한 한 끝까지 처리한다.

1. 요청을 작은 구현 단위로 해석한다.
2. 관련 코드와 설정을 확인한다.
3. 최소 변경으로 기능을 구현한다.
4. 가능한 가장 작은 검증을 실행한다.
5. 변경 내용을 스스로 리뷰한다.
6. 리뷰에서 확인된 저위험 개선 1건은 필요하면 즉시 반영한다.
7. 결과, 검증, 남은 리스크를 `Harness/cycles/`와 `Harness/state.md`에 남긴다.

단, Public API, Blueprint 노출 이름, `*.Build.cs`, 저장 데이터, 여러 모듈에 걸친 구조 변경이 필요해지면 먼저 영향 범위를 정리하고 신중하게 진행한다.

안전한 변경만 고르다가 사용자 목표와 무관한 로그 문구, 한글 깨짐, 주석, 포맷 정리만 반복해서는 안 된다. 자동 개선은 기능 완성도를 높이기 위한 수단이며, 기능 구현을 대신하지 않는다.

## 사이클 지시

사용자가 "사이클", "반복", "최대 N회"처럼 반복 작업을 지시하면 한 번의 응답 안에서 가능한 범위까지 아래 루프를 반복한다.

예: "검색창을 만들어줘. 사이클은 최대 10회로."

- 최대 횟수가 있으면 그 횟수를 넘지 않는다.
- 최대 횟수가 없으면 기본 1회만 수행한다.
- 각 사이클은 `구현 또는 개선 -> 검증 -> 자기 리뷰 -> 기록`으로 끝난다.
- 다음 사이클은 직전 사이클의 리뷰 결과나 남은 작업이 있을 때만 진행한다.
- 기능이 성공 기준을 만족하고 더 진행할 안전한 후보가 없으면 최대 횟수 전이라도 멈춘다.
- `HOLD` 또는 `STOP`에 해당하는 리스크가 나오면 반복을 멈추고 이유를 기록한다.
- 한 사이클에서 너무 큰 리팩터링으로 번지면 진행하지 않고 별도 작업 후보로 남긴다.
- 반복 사이클의 우선순위는 `기능 동작 -> 검증 실패 수정 -> 사용자 경험 개선 -> 안정성 보강 -> 기록 정리` 순서다.
- 로그 문구, 한글 깨짐, 주석, 포맷 정리는 디버깅이나 사용자 요청에 직접 필요할 때만 사이클 후보로 삼는다.

사이클 기록은 가능하면 오늘 날짜의 `Harness/cycles/YYYY-MM-DD.md`에 누적한다.

## 사용자 지시 해석

- 사용자가 아래처럼 짧게 지시해도 충분히 작업을 시작할 수 있다고 보고 처리한다.
- 예: `"인벤토리 UI 만들어줘. 최대 6사이클."`
- 예: `"락온 기능 보강해줘. 4사이클 안에서 끝까지."`
- 예: `"지금 에러 고쳐줘. 빌드 통과까지 반복해."`
- 기능명, 최대 사이클 수, 성공 기준이 보이면 추가 질문보다 바로 구현과 검증을 우선한다.
- 성공 기준이 불명확하면 현재 문맥에서 가장 자연스러운 기준을 짧게 정리한 뒤 진행한다.
- 반복 작업 중에는 이전 사이클의 긴 대화보다 `Harness/state.md`, `Harness/next.md`, 오늘 날짜의 `Harness/cycles/YYYY-MM-DD.md`를 우선 신뢰한다.

## 작업 규칙

- 현재 요청과 직접 관련된 파일만 수정한다.
- Unreal 프로젝트 구조와 기존 코드 스타일을 우선 존중한다.
- 전역 리팩터링, 파일 이동, 대규모 이름 변경은 요청 없이는 하지 않는다.
- 불필요한 포맷 변경이나 넓은 정리는 하지 않는다.
- 추측보다 확인한 코드, 문서, 로그, 실행 결과를 우선한다.
- 확인되지 않은 내용은 사실처럼 단정하지 않는다.
- 사용자가 만든 변경이나 관련 없는 생성물을 되돌리지 않는다.

## 언어 규칙

- 사용자가 한국어로 요청하면 응답은 한국어로 한다.
- Harness 기록 파일은 기본적으로 한국어로 작성한다.
- `Harness/state.md`, `Harness/next.md`, `Harness/cycles/YYYY-MM-DD.md`는 한국어를 기본 언어로 한다.
- `Harness/reviews/`의 리뷰 컨텍스트는 가능하면 한국어로 작성한다.
- 외부 AI 리뷰 원문은 원문 그대로 `Harness/reviews/`에 저장할 수 있다.
- 외부 AI 리뷰를 `cycles/`, `state.md`, `next.md`에 요약할 때는 한국어로 요약한다.
- 코드 식별자, 파일명, 클래스명, 함수명, 명령어, 로그 원문, 에러 메시지는 원문 언어를 유지한다.
- Windows PowerShell에서 Harness의 한국어 문서를 읽거나 쓸 때는 가능한 한 `-Encoding UTF8`을 명시한다.

## Git 규칙

- Git 저장소라면 작업 시작 전 `git status --short`로 현재 변경 상태를 확인한다.
- 아직 Git 저장소가 아니면 이 사실을 짧게 알리고, 템플릿 파일 수정은 계속 진행할 수 있다.
- 사용자가 만든 변경이나 현재 요청과 무관한 변경은 되돌리거나 함께 정리하지 않는다.
- `Binaries/`, `Intermediate/`, `Saved/`, `DerivedDataCache/` 같은 생성물은 커밋 대상으로 다루지 않는다.
- `Content/` 아래 `*.uasset`, `*.umap`을 수정한 경우 어떤 에셋이 왜 바뀌었는지 작업 기록에 남긴다.
- Redirector 정리, 에셋 이동, 대량 rename은 요청이 없으면 진행하지 않는다.
- `.gitattributes`에서 `*.uasset`, `*.umap`을 Git LFS로 추적한다면, 저장소가 실제로 LFS를 사용하도록 초기화되었는지 먼저 확인한다.
- 새 바이너리 에셋이나 대용량 원본 리소스를 추가할 때는 LFS 추적 대상인지 먼저 확인한다.
- 커밋은 사용자가 요청한 경우에만 수행한다.
- 브랜치 생성, rebase, force-push, history rewrite는 사용자가 명시적으로 요청한 경우에만 한다.
- 작업 후 `git diff --stat` 또는 필요한 파일 diff를 확인해 변경 범위가 요청과 일치하는지 검토한다.

## 기능 구현 루프

기능 작업은 아래 순서를 기본값으로 따른다.

1. 요청, 범위, 검증 방법을 짧게 정리한다.
2. 필요한 파일만 읽고 기존 패턴을 파악한다.
3. 구현 전 리스크를 짧게 확인한다.
4. 기능을 최소 단위로 구현한다.
5. 컴파일, 스크립트, 설정 검증 중 가능한 가장 작은 명령을 실행한다.
6. 변경 파일을 리뷰 관점으로 다시 읽고 Unreal 리스크를 점검한다.
7. 리뷰 결과가 안전한 경우에만 작은 개선을 반영한다.
8. 자동 검증으로 확인하지 못한 PIE, 입력, HUD, 카메라, 플레이 감각 항목은 수동 검증 필요로 기록한다.

자동 개선은 기능 구현을 대체하지 않는다. 먼저 요청된 기능이 동작 가능한 상태여야 하며, 개선은 그 다음에 작은 범위로만 수행한다.

반복 사이클 지시가 있는 경우 첫 사이클은 요청된 기능 구현을 우선하고, 이후 사이클은 검증 실패 수정, 자기 리뷰에서 발견한 안전한 개선, 누락된 기록 정리 순서로 진행한다.

사이클 후보를 고를 때는 가장 안전한 변경이 아니라 사용자 목표에 가장 가까워지는 안전한 변경을 고른다. 예를 들어 "검색창을 만들어줘"라는 요청에서는 검색 UI, 입력 처리, 검색 결과 연결, 빈 검색어 처리, 검증 가능한 테스트 상태를 로그 문구 정리보다 우선한다.

사이클별 기록은 길게 쓰지 않는다.

- `cycles/YYYY-MM-DD.md`에는 시도, 결과, 다음 액션만 짧게 남긴다.
- 최신 상태, 현재 동작, 확정된 사실은 `Harness/state.md`로 올린다.
- 다음 작업 후보, 미해결 문제, 나중에 볼 항목은 `Harness/next.md`로 분리한다.
- 같은 설명을 `state.md`, `next.md`, `cycles/`에 중복해서 길게 쓰지 않는다.

## 자동 개선 사이클

별도 개선 요청이나 작업 후 품질 개선이 필요하면 아래 판정 흐름을 따른다.

1. `EXPLORE`: 관련 범위에서 저위험 개선 후보를 찾는다.
2. `REVIEW`: 후보가 빌드, Blueprint, API, 런타임 의미를 깨지 않는지 판정한다.
3. `IMPROVE`: `SAFE` 또는 `SAFE_WITH_NOTE` 후보 1건만 반영한다.
4. `VERIFY`: 가능한 가장 작은 검증을 다시 실행한다.
5. `RECORD`: 결과를 `Harness/cycles/YYYY-MM-DD.md`에 남긴다.

판정값은 `SAFE`, `SAFE_WITH_NOTE`, `HOLD`, `STOP` 중 하나로 기록한다.

- `SAFE`: 바로 적용 가능하다.
- `SAFE_WITH_NOTE`: 적용 가능하지만 남은 주의점이 있다.
- `HOLD`: 추가 정보나 수동 확인 전에는 적용하지 않는다.
- `STOP`: 현재 범위에서 진행하지 않는다.

개선 후보는 아래 기준을 함께 만족해야 한다.

- 현재 요청 또는 성공 기준과 직접 관련이 있다.
- 실제 동작, 검증 가능성, 안정성, 사용성 중 하나를 높인다.
- 단순 로그 문구, 한글 깨짐, 주석, 포맷 정리만으로 끝나지 않는다. 단, 그 문제가 현재 기능 검증이나 런타임 진단을 막고 있으면 허용한다.

## Unreal 규칙

- `Source/`와 `Plugins/`는 작업 목적과 직접 관련될 때만 수정한다.
- `Config/`는 입력, 맵, 모듈, 빌드 설정 확인이 필요할 때만 수정한다.
- `Content/`, `Build/`, 생성 파일은 꼭 필요한 경우에만 수정한다.
- Blueprint 호환성이 깨질 수 있는 `UFUNCTION`, `UPROPERTY`, 공개 함수명, 공개 변수명 변경은 신중하게 한다.
- Public API 변경은 꼭 필요한 경우에만 한다.
- `*.Build.cs` 수정은 꼭 필요한 경우에만 하고, module dependency 영향을 함께 확인한다.
- include 경로, 모듈 경계, Public/Private 헤더 위치를 함께 점검한다.
- UObject, Actor, ActorComponent, Delegate 사용 시 null 체크와 라이프사이클을 확인한다.
- Delegate를 바인딩한 경우 해제 시점과 중복 바인딩 가능성을 함께 확인한다.

## 검증

- 가능한 가장 작은 빌드 또는 검증 명령을 실행한다.
- `Harness/scripts/verify_project.py` 통과만으로 성공 판정을 내리지 않는다. C++나 모듈 변경이 있으면 가능한 범위에서 실제 빌드 검증을 우선한다.
- 빌드 검증이 필요하면 `Harness/scripts/build_verify.ps1` 같은 표준 스크립트를 우선 사용하고, 실패 시 실패 지점과 명령을 함께 기록한다.
- C++ 수정 후에는 include 누락, module dependency, Blueprint 영향 가능성을 다시 점검한다.
- 에디터가 켜진 상태에서 C++ 검증이 DLL 잠금 또는 hot reload 산출물 문제로 실패하면, 에디터 종료 후 `-NoHotReload` 정식 빌드를 우선 재시도한다.
- 레벨, 입력, 에셋, HUD, 카메라, 애니메이션, 플레이 감각 같은 런타임 동작은 자동 검증만으로 충분하지 않을 수 있으므로 필요한 경우 PIE 수동 확인 필요성을 기록한다.
- 실행하지 못한 테스트나 검증이 있으면 이유를 남긴다.

## 작업 후

1. 변경한 내용과 검증 결과를 짧게 정리한다.
2. 오늘 날짜의 `Harness/cycles/YYYY-MM-DD.md`가 있으면 작업 기록을 추가한다.
3. `Harness/state.md`가 있으면 다음 작업자가 이어받을 수 있게 갱신한다.
4. 남은 리스크나 다음 액션이 있으면 명확히 적는다.
